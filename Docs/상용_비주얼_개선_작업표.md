# 상용 비주얼 개선 작업표

작성일: 2026-06-08

## 목적

`돌봄지원 심사소`는 `Papers, Please`를 주요 벤치마크로 삼았지만, 현재 화면 첫인상은 아직 실제 판매되는 완성품보다 기능을 많이 붙인 개발 빌드에 가깝다.

이 문서는 기존의 낙관적인 완성도 평가를 대체하지 않는다. 대신 현재 빌드가 왜 조잡해 보이는지 냉정하게 정리하고, 앞으로 어떤 순서로 비주얼과 화면 경험을 고쳐야 하는지 기록한다.

## 현재 판정

- 콘텐츠와 기능의 폭은 충분하지만, 화면의 질서와 조작 감각은 아직 상용 게임 기준에 못 미친다.
- 자동 QA의 `오버플로 0건`, `스모크 통과`는 화면이 깨지지 않는다는 증거일 뿐, 완성품처럼 보인다는 증거가 아니다.
- `Papers, Please`와의 격차는 기능 수가 아니라 정보 압축, 물리적 서류 감각, UI 일관성, 판단 루프의 선명함에서 발생한다.
- 지금 필요한 것은 기능 추가보다 화면을 덜어내고, 중요한 행동을 먼저 보이게 만들고, 문서/도장/기준표가 실제 사무실 물건처럼 느껴지게 만드는 작업이다.

## 왜 조잡해 보이는가

### 1. 정보 밀도가 너무 높다

한 화면에 신청자 정보, 서류, 기준, 지표, 안내문, 결정 버튼, 단축키, 추천, 기록성 텍스트가 동시에 노출된다. 플레이어는 무엇을 먼저 봐야 하는지보다 모든 것을 읽어야 한다는 압박을 먼저 받는다.

상용 게임 화면은 정보가 많아도 핵심 행동이 먼저 보인다. 현재 심사 화면은 핵심 서류 판단보다 주변 패널과 설명 텍스트가 더 크게 느껴진다.

### 2. UI 장식이 핵심 사물을 가린다

버튼, 패널, 라벨, 테두리, 그림자, 반투명 박스가 모두 강한 존재감을 가진다. 그 결과 신청서, 소득증명, 도장, 기준표 같은 게임의 핵심 사물이 화면에서 주인공처럼 보이지 않는다.

특히 작은 라벨을 별도 장식 박스에 계속 넣는 방식은 `신청서`, `소득증명` 같은 짧은 텍스트에서도 어색함을 반복해서 만든다.

### 3. 화면 미감이 한 규칙으로 묶이지 않는다

배경은 사실적인 사무실 톤이고, 가족 이미지는 일러스트/생성 이미지 톤이며, UI 패널은 게임식 장식 테두리이고, 버튼은 또 다른 크롬 스타일을 가진다. 각각은 의도가 있지만 한 화면에서 섞이면 같은 게임의 한 시스템처럼 보이지 않는다.

색상도 승인, 보류, 위험, 예산, 민원, 기록, 설정의 의미가 화면마다 충분히 엄격하게 고정되어 있지 않아 시각적 언어가 흐려진다.

### 4. 텍스트가 게임 문장보다 기능 설명에 가깝다

현재 문구는 많이 다듬었지만, 여전히 플레이어에게 행동을 시키는 짧은 문장보다 시스템 기능을 설명하는 문장이 많다. 결과적으로 화면이 게임 속 사무실 장비가 아니라 관리 도구 또는 QA 대시보드처럼 보일 때가 있다.

`Papers, Please`는 긴 설명보다 여권, 규정, 도장, 벌금, 줄 서 있는 사람의 압박으로 말한다. 우리 게임도 수치를 다 보여주기보다 지금 판단에 필요한 한 줄과 물건 배치를 먼저 사용해야 한다.

### 5. 절대 좌표 기반 레이아웃의 한계가 반복된다

UI가 많은 부분에서 고정 위치와 고정 크기로 배치되어 작은 해상도, 긴 한글 라벨, 큰 글자 모드에서 정렬 문제가 재발하기 쉽다. 자동 축소와 마스크로 막아도 근본적으로 공간이 부족한 영역은 계속 어색해 보인다.

작은 텍스트를 박스 안에 억지로 넣는 방식보다, 라벨 체계 자체를 넓고 안정적인 문서 탭/캡션 구조로 바꾸는 것이 필요하다.

### 6. 심사 화면이 책상이 아니라 대시보드처럼 보인다

게임의 핵심은 신청서를 읽고, 증빙을 확인하고, 기준표와 대조한 뒤 도장을 찍는 것이다. 하지만 현재 화면은 여러 기능 패널이 책상 위에 겹쳐진 느낌이 강하다.

심사 화면은 `읽는 UI`가 아니라 `처리하는 책상`이어야 한다. 플레이어가 버튼을 누른 것이 아니라 서류를 처리했다는 감각을 받아야 한다.

### 7. 리포트와 정책 화면은 데이터는 많지만 결론이 늦다

정책 시뮬레이터와 최종 리포트는 좋은 기능이지만, 수치와 표가 먼저 보이고 결론이 늦게 보이는 구간이 있다. 상용 게임의 데이터 화면은 표를 보여주기 전에 플레이어가 기억할 결론, 손해, 다음 행동을 먼저 제시해야 한다.

## Papers, Please에서 배워야 할 기준

- 첫눈에 지금 해야 할 일이 보인다.
- 서류, 도장, 규정표가 실제로 손에 잡히는 물건처럼 느껴진다.
- 한 번의 판단 루프가 짧고 강하다.
- 텍스트보다 사물, 상태 변화, 벌점, 소리, 도장 반응이 먼저 말한다.
- 화면마다 같은 색, 같은 글꼴 위계, 같은 버튼 규칙을 유지한다.
- 플레이어가 읽은 정보가 즉시 판단 행동으로 이어진다.

## 앞으로 할 일

### P0. 심사 화면 전면 재정리

목표는 심사 화면을 기능 대시보드가 아니라 서류 처리 책상으로 보이게 만드는 것이다.

- 신청자, 신청서, 소득증명, 기준표, 도장 영역을 화면의 핵심 축으로 재배치한다.
- 현재 판단에 직접 필요하지 않은 보조 지표와 설명은 접거나 작은 보조 패널로 낮춘다.
- `신청서`, `소득증명`은 작은 장식 박스가 아니라 문서 탭 또는 고정 캡션 체계로 통일한다.
- 결정 버튼은 일반 버튼보다 도장 트레이처럼 보이도록 정리한다.
- 판단 직후 피드백은 긴 설명 대신 `결과`, `근거`, `위험`, `다음 행동` 순서의 짧은 카드로 압축한다.

완료 기준:

- 첫 사례에서 3초 안에 읽을 서류와 찍을 도장이 보인다.
- 플레이어가 버튼 UI보다 서류와 도장을 먼저 인식한다.
- 1280x720에서도 작은 라벨이 억지로 끼워진 느낌이 없다.

### P0. UI 컴포넌트 규칙 정리

목표는 화면마다 다른 임시 스타일을 줄이고, 상용 UI처럼 반복 가능한 규칙을 만드는 것이다.

- 제목, 소제목, 본문, 라벨, 숫자, 버튼 텍스트의 크기와 색을 고정된 토큰으로 정한다.
- 패널, 카드, 버튼, 문서 라벨의 padding, line height, overflow 규칙을 통일한다.
- 작은 한글 라벨은 자동 축소에 기대지 않고 처음부터 충분한 폭과 높이를 확보한다.
- 같은 역할의 버튼은 같은 크기, 같은 색, 같은 위치 규칙을 따른다.
- 장식 테두리와 그림자는 줄이고, 핵심 정보 대비를 우선한다.

완료 기준:

- 새 UI를 만들 때 임의의 좌표/크기/색을 추가하지 않아도 된다.
- 굵은 글씨가 제목과 핵심 숫자에만 제한적으로 쓰인다.
- 버튼/라벨/카드의 정렬 문제가 개별 땜질로 반복되지 않는다.

### P0. 텍스트 압축

목표는 화면 문장을 기능 설명에서 플레이 문장으로 바꾸는 것이다.

- 긴 안내문은 `상황`, `판단 기준`, `행동` 중심의 짧은 문장으로 줄인다.
- 리포트와 정책 화면은 `결론 -> 이유 -> 다음 행동` 순서로 재작성한다.
- 내부 개발, 분석 도구, QA 산출물처럼 느껴지는 표현은 계속 제거한다.
- 버튼 라벨은 기능명이 아니라 플레이어 행동으로 쓴다.
- 반복되는 지표 설명은 화면마다 다시 쓰지 않고 시각화와 툴팁성 보조 문장으로 낮춘다.

완료 기준:

- 주요 화면의 첫 문단은 두 줄 이내로 읽힌다.
- 버튼 라벨만 봐도 다음 행동이 명확하다.
- 플레이어가 수치 설명을 읽기 전에 핵심 결론을 이해한다.

### P1. 메인 메뉴 재구성

목표는 첫 화면을 기능 모음이 아니라 게임 진입 화면으로 만드는 것이다.

- 새 캠페인, 이어하기, 기록, 설정의 우선순위를 명확히 나눈다.
- 저장 슬롯과 추천 회차 정보는 메인 행동을 방해하지 않는 위치로 낮춘다.
- 왼쪽 문서 패널의 설명문을 줄이고, 현재 플레이 상태를 더 짧게 보여준다.
- 배경 아트와 메뉴 UI가 서로 싸우지 않게 밝기와 대비를 재조정한다.

완료 기준:

- 처음 보는 사람이 3초 안에 `새 캠페인` 또는 `이어하기`를 찾는다.
- 메뉴가 QA/기능 목록이 아니라 게임의 첫 장면처럼 보인다.
- 보조 정보가 메인 버튼보다 먼저 튀어나오지 않는다.

### P1. 리포트와 정책 화면 리디자인

목표는 데이터 화면을 관리 도구가 아니라 게임 안의 분석 장비처럼 보이게 만드는 것이다.

- 최종 리포트는 이미지, 핵심 결과, 주요 지표, 다음 회차 목표 중심으로 다시 묶는다.
- 정책 화면은 수치 나열보다 비교 막대, 추천 결론, 위험 경고가 먼저 보이게 한다.
- 표와 문장 덩어리를 줄이고 카드 수를 제한한다.
- 같은 지표 색상은 심사 화면, 정책 화면, 리포트에서 동일하게 유지한다.

완료 기준:

- 정책 화면을 열자마자 어떤 기준이 위험하고 어떤 기준이 유리한지 보인다.
- 리포트는 읽기 전에 캠페인 결과의 감정과 방향이 먼저 전달된다.
- 데이터가 많아도 관리툴이 아니라 게임 결과 화면처럼 보인다.

### P1. 상용 비주얼 QA 도입

목표는 자동 스모크 테스트와 별개로 사람이 보는 완성도 기준을 만든다.

- 주요 화면 캡처를 `메인 메뉴`, `심사`, `서류 영역`, `정책`, `리포트`, `기록`, `저해상도`로 고정 수집한다.
- 각 화면을 첫인상, 정보 위계, 정렬, 색상, 글꼴, 상호작용 기대감 기준으로 1-5점 평가한다.
- `박스 밖으로 안 나감`이 아니라 `상용 게임처럼 보임`을 합격 기준으로 둔다.
- 3점 이하 항목은 다음 작업표에 바로 연결한다.

완료 기준:

- 핵심 화면 평균 4점 이상을 받을 때만 비주얼 정리 완료로 본다.
- 자동 QA 통과와 사람 눈 평가를 별도 증거로 기록한다.
- 캡처를 보고도 문제를 찾지 못하는 상황을 방지한다.

### P2. 모션, 사운드, 피드백 polish

목표는 화면의 정적인 조잡함을 줄이고, 판단 행위가 살아 있는 시스템처럼 느껴지게 만드는 것이다.

- 도장 찍기, 종이 넘김, 기준표 열기, 사건 발생, 저장 확인의 모션 규칙을 통일한다.
- 승인, 조건부, 보류, 추가조사, 거절의 소리와 색 반응을 더 분명하게 분리한다.
- 위험, 예산, 민원, 형평성 경고는 같은 색상 언어와 짧은 효과음으로 묶는다.
- 반복 피로를 줄이기 위해 과한 흔들림보다 짧고 건조한 사무실 반응을 유지한다.

완료 기준:

- 판단할 때마다 도장을 찍었다는 감각이 분명하다.
- 경고와 보상 피드백이 화면마다 같은 규칙으로 읽힌다.
- 연출이 과하지 않으면서도 현재보다 덜 정적인 느낌을 준다.

## 작업 원칙

- 기능 추가보다 화면 덜어내기를 우선한다.
- 자동 테스트 통과를 상용 비주얼 통과로 착각하지 않는다.
- 작은 라벨 문제를 개별 패딩으로만 고치지 말고, 라벨 체계 자체를 고친다.
- 개발자에게 유용한 정보와 플레이어에게 필요한 정보를 분리한다.
- `Papers, Please`를 복제하지 않는다. 그 게임의 압축, 물리감, 화면 질서만 우리 게임에 맞게 배운다.

## 다음 실행 순서

1. 현재 심사 화면 캡처를 기준으로 핵심/보조/불필요 정보를 색으로 분류한다.
2. `신청서`/`소득증명` 영역을 문서 탭 체계로 다시 설계한다.
3. 결정 버튼 5개를 도장 트레이 구조로 재배치한다.
4. 심사 화면의 보조 지표와 안내문을 접힘/축약 패널로 낮춘다.
5. 1280x720과 1920x1080 캡처를 비교해 사람 눈 기준으로 재평가한다.
6. 같은 규칙을 메인 메뉴, 정책 화면, 최종 리포트에 순차 적용한다.
