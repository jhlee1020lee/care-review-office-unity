# 돌봄지원 심사소 확장 개발 GOAL

작성일: 2026-05-31

이 문서는 새 Codex 세션에서 그대로 goal/context로 읽히도록 만든 개발 로드맵이다. 목표는 `돌봄지원 심사소`를 단순 Unity 게임이 아니라, 교수님 피드백과 2026-05-29 수업 내용을 반영한 고완성도 기말 프로젝트로 확장하는 것이다.

## 1. 최상위 목표

`돌봄지원 심사소`는 합성 가족지원 신청 사례를 input으로 받아, 플레이어와 여러 AI 에이전트의 판단을 로그로 남기고, 그 판단 경향을 대시보드와 보고서로 분석하는 게임형 의사결정 실험 플랫폼이어야 한다.

발표에서 한 문장으로 이렇게 설명할 수 있어야 한다.

> 합성 가족지원 신청 사례를 게임 안에서 심사하게 만들고, 플레이어와 여러 AI 에이전트의 판단 로그를 수집해, 가족 유형·위험도·서류 완성도·예산 상황에 따라 승인/거절/추가조사 판단이 어떻게 달라지는지 분석하는 프로젝트입니다.

## 2. 교수님 피드백에서 지켜야 할 기준

교수님 말씀을 기준으로, 이 프로젝트는 다음 조건을 만족해야 한다.

1. 게임/서비스형 output 프로젝트는 수업 취지에 맞다.
   - 단순 분석 리포트보다 난이도가 높다.
   - 실제 작동하는 결과물과 판단/분석 로직을 보여주면 강점이 된다.

2. 많이 하더라도 얕게 하지 않는다.
   - 여러 기능을 넣되 모든 기능이 판단, 로그, 분석, 윤리, 스토리텔링 중 하나에 연결되어야 한다.
   - 기능 수가 많아도 발표에서 왜 필요한지 설명할 수 있어야 한다.

3. 합성 데이터의 한계를 반드시 밝힌다.
   - 실제 가족지원 심사 데이터가 아니다.
   - 실제 정책 효과나 실제 가족 유형의 특성을 일반화하지 않는다.
   - 목적은 의사결정 경향을 탐색하는 것이다.

4. 분석은 Research Question에서 출발한다.
   - 먼저 무엇을 알고 싶은지 정한다.
   - 그다음 어떤 로그와 대시보드가 필요한지 설계한다.

5. AI 사용 기록을 남긴다.
   - 사용한 모델/도구/날짜/목적을 기록한다.
   - 이미지 생성, 코드 구현, 에이전트 시뮬레이션, QA 자동화 과정을 제출물에 포함한다.

## 3. 사용자의 강점과 개발 전략

사용자는 ChatGPT Pro 요금제를 사용하고 있고, 다른 학우들의 Claude Pro 사용 환경보다 작업 한도와 이미지 생성 활용 여지가 넉넉하다. 따라서 이 프로젝트는 최소 MVP에 머물지 않고, 기능을 최대한 많이 구현하는 방향으로 간다.

단, 기능 확장의 원칙은 다음과 같다.

- 장식 기능보다 판단 실험과 분석 기능을 우선한다.
- 게임 기능은 반드시 로그와 대시보드로 회수한다.
- 이미지 생성은 UI, 사례 파일, 상담 메모, 도장, 배경, 이벤트 화면 등에 적극 사용한다.
- 넉넉한 한도는 구현뿐 아니라 반복 검증, 에이전트 시뮬레이션, 보고서 작성, QA 자동화에 쓴다.

핵심 전략은 다음 문장이다.

> 많이 만들되, 모든 기능을 판단 실험과 로그 분석으로 회수한다.

## 4. 프로젝트의 핵심 구조

최종 시스템은 다음 흐름을 가져야 한다.

1. 합성 가족 사례 제시
2. 플레이어의 심사 판단
3. 판단 근거 선택
4. 예산/위험/형평성/민원/안정도 지표 변화
5. 플레이어 선택 로그 저장
6. AI 에이전트 자동 판단
7. 에이전트별 판단 로그 저장
8. 대시보드 분석
9. 최종 리포트/발표자료/AI 사용 로그로 정리

Input:

- 합성 가족 사례
- 가족 유형
- 위험도
- 서류 완성도
- 예산 상황
- 신청 지원 종류
- 운영 기준

Process:

- 플레이어 판단
- AI 에이전트 판단
- 권장 판단과 비교
- 결과 지표 반영

Output:

- 플레이어 로그
- 에이전트 로그
- 판단 성향 분석
- 권장 판단 일치율
- 위험 신호 누락 사례
- 가족 유형/위험도/예산별 판단 경향
- 대시보드와 보고서

## 5. Research Questions

보고서와 대시보드의 중심 RQ는 아래 3개로 둔다.

1. 가족 유형, 위험 신호, 서류 완성도, 예산 상황에 따라 심사 판단은 어떻게 달라지는가?
2. 서로 다른 성향의 AI 에이전트는 같은 사례를 어떻게 다르게 판단하는가?
3. 플레이어 판단과 기준 기반 권장 판단은 어떤 상황에서 어긋나는가?

확장 RQ:

- 예산 압박이 커질수록 승인율이 낮아지는가?
- 위험 신호가 높은 사례에서 추가조사/조건부 승인 비율이 높아지는가?
- 서류 완성도가 낮은 사례에서 에이전트별 판단 차이가 커지는가?
- 특정 가족 유형이 특정 에이전트에게 더 엄격하게 판단되는가?
- 플레이어는 예산, 위험, 서류, 형평성 중 무엇을 가장 자주 우선하는가?

## 7. 개발 단계별 로드맵

### Phase 1. MVP 루프 안정화

목표:

- 현재 구현된 게임 루프가 처음부터 끝까지 끊기지 않게 만든다.

작업:

- 새 캠페인 시작부터 최종 리포트까지 수동 플레이 검증
- 5개 이상 사례를 플레이해 로그 생성 확인
- 판단 직후 피드백 확인
- 최종 리포트/에이전트 분석 화면 확인
- CSV/JSON export 확인
- 핵심 스크린샷 5장 생성

완료 기준:

- `input -> 판단 -> 로그 -> 분석 output`이 한 번에 설명된다.
- 실행 파일, 스크린샷, 샘플 로그가 존재한다.

### Phase 2. 판단 시스템 강화

목표:

- 플레이어 선택이 단순 버튼 클릭이 아니라 다기준 의사결정처럼 보이게 만든다.

추가 기능:

- 판단 전 근거 선택 강화
- 선택지별 예상 결과 미리보기 강화
- 판단 후 권장 판단과의 차이 설명
- 위험 신호 놓침 경고
- 예산 절감과 지원 누락의 trade-off 표시
- 플레이어 판단 성향 점수 추가

최종 리포트에 들어갈 항목:

- 승인율
- 추가조사율
- 거절율
- 권장 판단 일치율
- 위험 사례 누락 수
- 예산 절감 지표
- 형평성 지표
- 플레이어 심사관 유형

### Phase 3. 사례 콘텐츠 확장

목표:

- 사례 수만 늘리는 것이 아니라, 분석 가능한 유형 다양성을 만든다.

추천 사례 유형:

- 한부모 가정
- 조손 가정
- 다문화 가정
- 장애 아동 포함 가정
- 돌봄 공백 가정
- 긴급 위기 가정
- 서류 미비 가정
- 소득은 낮지 않지만 위험 신호가 큰 가정
- 예산 압박 상황에서 애매한 사례
- 민원 가능성이 높은 사례

사례별 구성 요소:

- 가족 프로필 카드
- 신청 사유서
- 상담 메모
- 현장 방문 요약
- 학교/어린이집/복지기관 의견
- 추가조사 시 공개되는 새 정보

목표 규모:

- 최소: 현재 사례 40개 정리
- 권장: 40개 유지하되 태그/데이터 필드 보강
- 확장: 60개까지 확장

### Phase 4. AI 에이전트 시스템 고도화

목표:

- 교수님이 긍정적으로 본 에이전트 롤플레이 로그를 프로젝트 핵심 강점으로 만든다.

기본 에이전트:

- 예산 민감형
- 위험 우선형
- 서류 엄격형
- 형평성 우선형
- 추가조사 선호형

고급 에이전트:

- 민원 회피형
- 아동 안전 최우선형
- 행정 절차 우선형
- 가족 회복 가능성 중시형
- 데이터 기반 권장형
- 초보 심사관형
- 경험 많은 심사관형

기능:

- 같은 사례를 여러 에이전트가 판단
- 에이전트별 판단 이유 생성
- 에이전트 간 disagreement 표시
- 플레이어와 에이전트 판단 유사도 계산
- 특정 사례에서 판단이 갈린 이유 설명

완료 기준:

- 발표에서 “같은 사례라도 기준이 다르면 판단이 달라진다”를 실제 로그로 보여준다.

### Phase 5. 대시보드 강화

목표:

- 게임이 분석 가능한 시스템임을 시각적으로 증명한다.

필수 차트:

- 에이전트별 판단 비율
- 가족 유형별 승인/거절/추가조사 비율
- 위험도별 판단 변화
- 서류 완성도별 판단 변화
- 예산 압박별 승인율 변화
- 플레이어와 권장 판단 일치율
- 플레이어와 에이전트 유사도

고급 차트:

- 판단 혼동 행렬
- 위험 신호 놓침 사례 목록
- 과잉 거절 가능 사례 목록
- 형평성 지표
- 에이전트 간 disagreement heatmap
- 사례별 판단 분포
- 플레이어 판단 성향 radar chart

Export 목표:

- `dashboard.html`
- `player_log.csv`
- `agent_log.csv`
- `summary.json`
- dashboard screenshots

### Phase 6. 게임성 강화

목표:

- 과제용 게임이지만 실제 플레이 경험이 있게 만든다.

추가 시스템:

- 하루 단위 심사 일정
- 제한 예산
- 긴급 사례 우선순위
- 후속 민원/감사 이벤트
- 승인 후 긍정/부정 결과
- 거절 후 위험 악화 가능성
- 추가조사로 새 정보 해금
- 운영 기준 변경 이벤트
- 최종 감사 리포트
- 엔딩 분기

엔딩 예시:

- 균형형 심사관
- 예산 절감형 심사관
- 위험 방어형 심사관
- 서류주의 심사관
- 과잉 지원형 심사관
- 고위험 누락 심사관

주의:

- 엔딩은 재미용으로만 두지 않는다.
- 엔딩은 플레이어 판단 로그와 성향 분석 결과로 결정되어야 한다.

### Phase 7. 이미지 생성 기반 UI/자산 강화

목표:

- 이미지 생성 역량을 적극 활용해 완성도 높은 과제물로 보이게 한다.

생성할 자산:

- 메인 메뉴 배경: 가족지원센터 사무실
- 심사 책상 배경
- 사례 파일 표지
- 승인/거절/추가조사 도장
- 상담 메모지 텍스처
- 가족 프로필 일러스트
- 알림/민원/감사 이벤트 이미지
- 최종 리포트 표지 이미지
- 튜토리얼 컷신 이미지
- 대시보드 섹션 배너
- 윤리 고지 화면 이미지

UI 방향:

- 복지 행정/데이터 분석 느낌
- 너무 장난스럽지 않게
- 문서 기반 심사 게임 분위기
- 진짜 제출 가능한 완성본처럼 보이게

### Phase 8. 윤리/합성 데이터 고지 시스템

목표:

- 2026-05-29 수업의 데이터 윤리 내용을 프로젝트 안에 반영한다.

게임 안에 넣을 것:

- 첫 실행 시 합성 데이터 고지
- 실제 심사 기준이 아님 고지
- 민감한 가족 사례를 낙인찍지 말라는 안내
- AI 판단은 참고용이라는 안내
- 최종 리포트에 윤리적 한계 자동 포함

보고서에 넣을 것:

- 합성 데이터 사용 이유
- 실제 개인정보 미사용
- 결과 일반화 금지
- AI 에이전트 판단의 한계
- 데이터 편향 가능성
- 대시보드 해석 주의

핵심 문장:

> 본 프로젝트는 합성 사례와 가상 에이전트 로그를 활용하므로, 분석 결과를 실제 가족지원 심사 현장에 일반화하지 않는다. 대신 게임형 시스템이 판단 로그를 생성하고, 그 로그를 통해 기준별 판단 차이를 탐색할 수 있는지를 검증하는 데 목적이 있다.

### Phase 9. AI 사용 로그 자동화

목표:

- 화면 밖으로 나간 UI 없음
