## 1. 프로젝트 한 줄 설명

`돌봄지원 심사소`는 가족지원센터 심사 담당자가 되어 여러 가족의 신청서, 증빙서류, 위험 신호를 검토하고 제한된 예산 안에서 지원 여부를 결정하는 의사결정 시뮬레이션 게임이다.

게임 자체가 최종 산출물이지만, 기말 프로젝트의 핵심은 단순히 게임을 만드는 데 있지 않다. 플레이어가 가족의 돌봄 취약성을 어떤 기준으로 판단하는지 로그로 남기고, 그 로그를 분석하여 돌봄지원 판단의 편향, 기준 충돌, 예산 압박 효과를 보여주는 것이 최종 목표다.

## 2. 기말 프로젝트로서의 핵심 질문

주요 연구질문:

> 가족 돌봄지원 심사 게임에서 플레이어는 소득 기준, 서류 완성도, 위험 신호, 예산 제약 중 무엇을 우선하여 취약성을 판단하는가?

세부 질문:

1. 서류가 부족하지만 위험도가 높은 사례는 얼마나 승인되는가?
2. 소득 기준을 약간 초과하지만 돌봄 부담이 큰 가족은 어떻게 판단되는가?
3. 예산이 줄어들수록 플레이어의 판단은 더 엄격해지는가?
4. 한부모, 조손, 특수교육, 노인 돌봄, 다문화 가족 유형별 승인율은 차이가 나는가?
5. AI가 설계한 권장 판단과 인간 플레이어 판단은 어디서 달라지는가?

## 4. 게임 구조

### 4.1 기본 루프

1. 신청자 가족 사례가 들어온다.
2. 플레이어는 가족 유형, 신청 내용, 서류 상태, 위험 신호를 확인한다.
3. 오른쪽 규정집과 현재 예산/지표를 함께 본다.
4. 플레이어는 판단 근거를 고른 뒤 승인, 조건부 승인, 보류, 추가조사, 거절 중 하나를 선택한다.
5. 선택에 따라 예산, 가족 안정도, 형평성, 누락 위험, 민원 위험이 변한다.
6. 판단과 근거가 함께 로그에 저장되고, 모든 사례를 처리하면 결과 리포트가 나온다.

### 4.2 의사결정 버튼

- 승인: 예산을 가장 많이 쓰지만 가족 안정도와 위험 완화 효과가 큼
- 조건부 승인: 일부 지원을 제공하고 위험을 부분적으로 낮춤
- 보류: 예산은 아끼지만 위험과 민원 가능성이 증가
- 추가조사: 즉시 지원은 아니지만 판단 보류보다 형평성 측면에서 나음
- 거절: 예산은 보존되지만 위험도가 높은 사례에서는 누락 위험이 크게 증가

### 4.3 관리 지표

- 남은 예산: 당일 심사에 쓸 수 있는 총 지원 자원
- 가족 안정도: 승인/조건부 승인으로 상승, 거절/보류로 하락
- 형평성 점수: 위험과 접근성 장벽을 고려한 판단을 할수록 상승
- 누락 위험: 위험 사례를 거절하거나 보류할수록 상승
- 민원 위험: 서류가 애매한데 무리하게 승인하거나, 긴급 사례를 거절할 때 상승

## 5. 시각적 방향

시각 콘셉트:

- 장르: 서류 심사형 사회 시뮬레이션
- 분위기: 따뜻하지만 긴장감 있는 가족지원센터 사무실
- 레퍼런스 구조: `Papers, Please`식 서류 심사 + `Death and Taxes`식 사례 판단 + 생성 이미지 기반 고품질 UI
- 색감: 어두운 목재 책상, 바랜 서류지, 청록색 규정집, 붉은 도장, 따뜻한 조명

이미지 생성 활용 계획:

- 가족 유형별 초상 카드
- 신청서/진단서/소득증명서/학교 의견서
- 도장, 도장패드, 서류철, 규정집
- 버튼, 탭, 경고 스티커, 알림 쪽지 같은 작은 UI 부품
- 결과 리포트용 장면 이미지
- 발표자료 대표 이미지
- 교수님께 보여줄 썸네일 또는 짧은 게임 소개 이미지

제작 원칙:

- 웬만한 시각 요소는 코드로 대충 그리지 않고 이미지 생성으로 먼저 만든다.
- Unity에서는 생성 이미지를 잘라 쓰고, 클릭 영역과 텍스트만 코드로 얹는다.
- 작은 UI 요소도 일관된 질감이 중요하므로 도장, 버튼, 종이 탭, 판정 라벨, 경고 아이콘을 별도 이미지 세트로 만든다.
- 단, 실제 숫자, 사례 텍스트, 분석 결과처럼 계속 바뀌는 정보는 Unity 텍스트로 처리한다.
- 생성 이미지가 너무 장식적으로 보이지 않도록 `가족지원센터 사무실`, `행정 서류`, `돌봄 심사`, `교육용 시뮬레이션` 톤을 유지한다.

코드로 처리할 요소:

- 실제 클릭 영역
- 텍스트 출력
- 지표 계산
- 로그 저장
- 결과 리포트
- 분석용 데이터 내보내기

### 6.2 2차 작업: 게임성 보강

목표:

단순 클릭형 데모에서 `판단 압박이 느껴지는 게임`으로 확장한다.

작업 목록:

2. 1일차 구조를 3일차 구조로 확장

4. 신청서 안에 명시 정보와 숨은 단서를 분리
5. 잘못된 판단을 반복하면 엔딩 리포트에서 성향을 더 강하게 보여줌
6. 사례별 권장 판단을 더 정교하게 조정
7. 선택 후 짧은 가족 후속 상황을 보여주는 카드 추가
8. 선택 직후 내 판단이 권장 판단과 비교해 예산·안정·위험을 어떻게 바꾸는지 표시

게임성을 강화할 때의 핵심은 빠른 반사신경이나 복잡한 조작이 아니다. 이 프로젝트에 맞는 재미는 `애매한 사례`, `부족한 예산`, `서류와 실제 위험의 충돌`, `내 선택이 누적되어 드러나는 결과`에서 나온다. 따라서 앞으로의 추가 기능도 전투, 이동, 퍼즐보다 심사 판단을 더 날카롭게 만드는 쪽에 집중한다.

예상 확장 사례:

- 특수교육 자녀의 치료기관 접근성 문제
- 한부모 야간노동과 방과후 돌봄 공백
- 조손가정의 보호자 건강 악화
- 성인 자녀의 노부모 돌봄 부담
- 다문화 가정의 행정 접근성 장벽
- 소득 기준은 넘지만 돌봄 비용이 큰 맞벌이 가족
- 서류는 완벽하지만 긴급도는 낮은 사례
- 서류는 부족하지만 안전 위험이 높은 사례

### 리스크 2: 사례가 임의적이라는 비판

대응:

### 리스크 3: 이미지 생성물이 과제의 핵심처럼 보일 수 있음

### 리스크 4: Unity 개발 시간이 과도하게 늘어날 수 있음

대응:

## 11. 상용화 확장 반영

2026-05-28 기준 프로토타입은 5일차 40개 사례, 3종 운영 기준, 캠페인 챌린지, 에이전트 분석, HTML 대시보드, 플레이테스트 패킷, 인게임 플레이테스트 회수 준비 체크, 지원 번들, 런타임 이슈 로그, 컨트롤러 단축 입력, 컨트롤러 포커스 하이라이트, 첫 실행 데이터/윤리 고지, 개인정보/데이터 처리 고지, 저장 백업/복구 QA, 로컬 데이터 삭제 QA, 데이터 소스 무결성 QA, Steam 마케팅 자산 감사 QA까지 확장되었다.

로컬 데이터 삭제 기능은 과제용 로그 분석과 상용 배포를 함께 고려한 장치다. 플레이어가 저장 슬롯, 로그, 세션 원본, 지원 번들, 성과/엔딩/캠페인 기록을 직접 지울 수 있게 하되, 2단계 확인과 게임 생성 파일만 삭제하는 안전장치를 둔다.

첫 실행 고지는 게임을 시작하기 전 합성 사례, 실제 개인정보 미사용, 로컬 저장, 서버 자동 업로드 없음, 실제 판단 비대체를 한 번 확인하게 해 과제 발표와 외부 플레이테스트에서 윤리적 설명을 명확히 한다.

데이터 소스 무결성 QA는 게임의 40개 사례와 5개 가상 플레이어 성향이 코드 내부 fallback이 아니라 외부 JSON 리소스에서 로드되는지 검증한다. 이 검증은 기말 프로젝트의 데이터 기반 산출물과 상용 릴리즈 후보의 콘텐츠 관리 신뢰도를 함께 보강한다.

Steam 마케팅 자산 감사 QA는 상점에 올릴 스크린샷, 캡슐 이미지, 트레일러 프레임, 업로드 후보 영상, 자산 zip에 누락이나 규격 오류, 로컬 절대경로 노출이 없는지 확인한다. 따라서 게임 실행본뿐 아니라 실제 판매 페이지에 들어갈 자료도 같은 릴리즈 후보 감사 흐름 안에서 관리한다.

저장 백업/복구 QA는 플레이어 진행 저장이 손상되었을 때 기본 저장을 그대로 실패 처리하지 않고 `.bak` 백업에서 복구되는지 검증한다. 과제용 실행본을 넘어 판매용 게임으로 볼 때, 진행 손실 가능성을 낮추는 안정성 근거로 릴리즈 후보 감사에 포함한다.

런타임 이슈 로그 QA는 외부 플레이테스터가 문제를 겪었을 때 단순한 말 설명이 아니라 익명화된 Warning/Error/Exception 기록과 지원 번들을 함께 보낼 수 있게 한다. 이는 Steam 비공개 브랜치와 저사양 PC 검증 단계에서 재현 정보를 확보하기 위한 안정성 장치다.

컨트롤러 단축 입력 QA는 Steam Deck이나 거실 PC처럼 마우스 없이 플레이하는 환경을 대비한 보강이다. 아직 완전한 패드 UI 포커스 시스템은 아니지만, 핵심 심사 판단과 주요 보조 화면을 기본 버튼으로 진행할 수 있는지 자동 검증해 외부 저사양/Steam 클라이언트 QA에서 확인할 범위를 줄인다.

인게임 플레이테스트 회수 준비 체크는 최종 리포트의 테스터 설문 화면에서 판단 35건 이상, 설문 저장, 로그 패킷 저장, 환경 진단 저장, 로컬 절대경로 미포함 여부를 `회수 준비` 한 줄로 보여준다. 이 장치는 실제 사람 플레이테스트를 받을 때 누락 파일을 줄이고, 기말 프로젝트 발표에서 게임이 단순 실행물이 아니라 분석 가능한 데이터 회수 도구라는 점을 설명하는 근거가 된다.

지원 번들 기반 회수 감사는 문제가 발생한 테스터가 `support_bundles/<bundleId>`만 보낸 경우에도 세션 원본, 설문, 환경 진단, manifest, 런타임 이슈 로그를 같은 감사 흐름에 올린다. 따라서 플레이테스트 회수 데이터와 버그 재현 정보가 분리되지 않고, 상용화 트리아지에서 실제 외부 완전 회수 수와 QA 샘플/지원 번들 검증 수를 구분해 볼 수 있다.

외부 게이트 증거 감사는 Steam 공개 출시를 막는 10개 조건을 별도 리포트로 분리한다. 자동으로 확인 가능한 플레이테스트 회수 수, AppID/DepotID placeholder 제거, SteamCMD preview 로그와 사람이 확인해야 하는 Store Presence, 업적, Steam 클라이언트 설치/재설치, 저사양 PC 검증 증거를 같은 표에 올려 릴리즈 후보 감사가 단순한 내부 빌드 통과가 아니라 실제 공개 준비도까지 추적하도록 한다.
