# 상용 비주얼 작업 방향

작성일: 2026-06-08
기준 문서: `Docs/상용_비주얼_개선_작업표.md`

## 핵심 방향

현재 목표는 기능을 더 붙이는 것이 아니라, 이미 있는 게임을 상용 제품처럼 보이게 만드는 것이다. `Papers, Please`를 따라 만든다는 뜻이 아니라, 그 게임이 가진 화면 질서, 물리적 서류 감각, 짧은 판단 루프를 우리 게임의 복지 심사 소재에 맞게 다시 설계한다.

앞으로의 작업은 아래 한 문장으로 판단한다.

> 플레이어가 화면을 보자마자 "무엇을 읽고, 무엇을 대조하고, 어떤 도장을 찍을지" 알 수 있어야 한다.

자동 스모크 테스트 통과는 보조 증거로만 본다. 최종 판단은 실제 캡처와 사람이 보는 첫인상으로 한다.

## 하지 말아야 할 방향

- 새 기능을 추가해서 조잡함을 덮지 않는다.
- 작은 글씨를 더 줄여서 박스 안에 넣는 식으로 해결하지 않는다.
- 자동 축소, 마스크, 임시 padding으로 같은 라벨 문제를 반복해서 고치지 않는다.
- 모든 수치와 설명을 한 화면에 보여주려 하지 않는다.
- QA, 분석, 개발 산출물처럼 보이는 표현을 플레이어 화면에 남기지 않는다.
- `Papers, Please`의 픽셀아트나 정치 소재를 복제하지 않는다.

## 1차 목표

1차 작업의 목표는 전체 게임 리디자인이 아니라, 가장 많이 보이는 화면의 상용 첫인상을 끌어올리는 것이다.

우선순위는 다음 순서로 둔다.

1. 심사 화면
2. 신청서/소득증명 서류 영역
3. 결정 버튼과 도장 영역
4. 메인 메뉴
5. 정책/기준표 화면
6. 최종 리포트

첫 작업은 반드시 심사 화면에서 시작한다. 이 화면이 여전히 개발 빌드처럼 보이면 메뉴나 리포트만 고쳐도 게임 전체 인상은 바뀌지 않는다.

## 심사 화면 작업 방향

심사 화면은 앞으로 `정보 대시보드`가 아니라 `서류 처리 책상`으로 설계한다.

### 화면 구조

- 왼쪽: 신청자와 기본 사건 식별 정보
- 중앙: 신청서와 소득증명, 핵심 증빙
- 오른쪽: 기준표, 운영 기준, 현재 압박 지표
- 하단: 도장 트레이와 판단 버튼
- 보조 정보: 접힘 패널 또는 작은 메모 형태로 낮춤

현재 화면에서 모든 것을 동시에 보여주려는 구조를 줄이고, 한 사례 판단에 필요한 순서를 화면 배치로 강제한다.

### 서류 영역

- `신청서`, `소득증명`은 작은 장식 라벨이 아니라 문서 자체의 탭 또는 캡션으로 표시한다.
- 두 라벨은 같은 폭, 같은 높이, 같은 기준선으로 배치한다.
- 라벨이 들어갈 공간을 먼저 정하고 그 안에 텍스트를 넣는다.
- 라벨 주변 장식선, 그림자, 반투명 배경이 텍스트보다 강하게 보이면 실패로 본다.
- 서류마다 제목, 본문, 상태 메모의 크기 차이를 명확히 둔다.

### 도장 영역

- 결정 버튼 5개는 일반 UI 버튼보다 도장 트레이처럼 보여야 한다.
- 승인, 조건부, 보류, 추가조사, 거절의 색은 명확하지만 과하게 튀지 않게 한다.
- 숫자 단축키는 보조 정보로 낮추고, 결정명과 도장 형태가 먼저 보이게 한다.
- 결정 직후에는 도장, 짧은 효과음, 한 줄 결과가 먼저 나오고 긴 설명은 뒤로 간다.

### 보조 정보

- 예산, 안정, 형평, 누락, 민원은 항상 같은 위치와 색을 사용한다.
- 현재 판단에 직접 필요하지 않은 추천/기록/상세 설명은 축약한다.
- 모든 보조 패널은 "지금 판단에 쓰는가" 기준으로 남기거나 낮춘다.

## 메인 메뉴 작업 방향

메인 메뉴는 기능 목록이 아니라 게임의 첫 장면이어야 한다.

- 가장 중요한 행동은 `새 캠페인 시작`과 `이어하기`다.
- 기록, 성과, 설정, 슬롯, 추천 정보는 보조 그룹으로 묶는다.
- 왼쪽 문서 패널은 세계관 설명보다 현재 플레이 상태와 다음 행동을 짧게 보여준다.
- 버전, 내부 상태, 세부 추천 문구는 작게 낮추거나 제거한다.
- 배경의 책상과 메뉴 패널이 서로 경쟁하지 않도록 메뉴 배경 대비를 정리한다.

합격 기준:

- 처음 보는 사람이 3초 안에 시작 버튼을 찾는다.
- 버튼이 너무 많다는 인상을 주지 않는다.
- 화면이 기능 런처가 아니라 사무실 안에서 근무를 시작하는 장면처럼 보인다.

## 정책/기준표 작업 방향

정책 화면은 숫자를 보여주는 표가 아니라 판단을 돕는 기준 장비여야 한다.

- 첫 줄에 현재 기준의 결론을 보여준다.
- 비교 막대와 추천 결론을 수치 표보다 먼저 배치한다.
- 정책별 차이는 색상, 막대 길이, 짧은 판정 문구로 읽히게 한다.
- 긴 사례 설명과 결과 리스트는 접거나 오른쪽 보조 영역으로 낮춘다.
- 심사 화면의 지표 색상과 같은 색 규칙을 유지한다.

합격 기준:

- 화면을 열자마자 어떤 기준이 위험하고 어떤 기준이 유리한지 보인다.
- 플레이어가 정책 화면을 보고 현재 사례 판단을 바꿀 이유를 얻는다.
- 관리툴 표가 아니라 게임 안의 기준표처럼 보인다.

## 최종 리포트 작업 방향

최종 리포트는 모든 데이터를 나열하는 화면이 아니라, 플레이어의 캠페인을 기억하게 만드는 화면이어야 한다.

- 큰 결과 이미지 또는 엔딩 카드가 첫인상을 담당한다.
- 핵심 결과, 주요 지표, 재검토 후보, 다음 회차 목표 순서로 읽히게 한다.
- 긴 분석 문장은 줄이고, 핵심 변화와 플레이어 선택의 결과를 먼저 보여준다.
- 표와 로그는 보조 상세로 낮춘다.
- 다음 회차를 시작하고 싶게 만드는 목표를 한 줄로 남긴다.

합격 기준:

- 리포트를 읽기 전에 캠페인 분위기와 결과가 먼저 느껴진다.
- 가장 중요한 수치 3-5개만 즉시 보인다.
- 다음 회차에서 무엇을 다르게 해볼지 보인다.

## UI 시스템 작업 방향

개별 화면을 고치기 전에 반복 규칙을 정해야 한다. 같은 문제가 계속 재발하는 이유는 UI 기본 규칙이 약하기 때문이다.

### 텍스트 계층

- 대제목: 화면 정체성을 표시하는 용도만 사용
- 소제목: 카드나 문서 구역의 제목
- 본문: 사례 설명과 결과 설명
- 라벨: 문서명, 지표명, 버튼 보조 문구
- 숫자: 지표와 변화량

각 계층은 크기, 색, 굵기를 고정한다. 굵은 글씨는 대제목, 버튼명, 핵심 숫자에만 제한한다.

### 패널 계층

- 종이 패널: 신청서, 리포트, 기준표
- 어두운 패널: 상태 메모, 경고, 보조 설명
- 버튼 패널: 실제 클릭 가능한 명령
- 장식 패널: 가능하면 줄이고, 정보와 경쟁하지 않게 한다.

패널 안에 패널을 계속 중첩하지 않는다. 중첩이 필요하면 안쪽 패널은 장식을 줄이고 여백으로 구분한다.

### 색상 규칙

- 승인/지원: 낮은 채도의 녹색
- 조건부/주의: 노란 갈색
- 보류/중립: 회색
- 추가조사/분석: 청록 또는 청회색
- 거절/위험: 낮은 채도의 붉은색
- 예산: 금색/갈색 계열
- 민원/위험 경고: 붉은색 계열

색은 장식이 아니라 의미로 사용한다. 같은 의미는 모든 화면에서 같은 색으로 유지한다.

## 작업 순서

### 1단계: 캡처 기준선 만들기

- 현재 메인 메뉴, 심사 화면, 정책 화면, 리포트 화면을 1280x720과 1920x1080으로 캡처한다.
- 각 화면에서 핵심 정보, 보조 정보, 불필요 정보, 장식 과다 영역을 표시한다.
- 자동 QA 통과 여부와 별개로 사람이 보는 문제를 적는다.

### 2단계: 심사 화면 와이어 정리

- 코드 수정 전에 심사 화면의 새 배치 원칙을 문서 또는 스케치로 확정한다.
- 신청자, 서류, 기준표, 도장 트레이의 위치와 크기를 먼저 정한다.
- 보조 정보는 어디에 접거나 낮출지 정한다.

### 3단계: 서류/도장 영역 구현

- `신청서`/`소득증명` 라벨 구조를 문서 탭/캡션 체계로 교체한다.
- 결정 버튼 5개를 도장 트레이처럼 재구성한다.
- 판단 직후 피드백을 짧은 결과 카드로 정리한다.

### 4단계: 사람 눈 QA

- 1280x720, 1920x1080 캡처를 다시 찍는다.
- 첫인상, 정보 위계, 정렬, 글꼴, 색상, 조작 기대감에 1-5점을 준다.
- 평균 4점 미만이면 다음 화면으로 넘어가지 않는다.

### 5단계: 메인 메뉴와 데이터 화면 적용

- 심사 화면에서 정한 텍스트 계층과 색상 규칙을 메인 메뉴에 적용한다.
- 정책 화면과 리포트 화면도 같은 패널/색상/텍스트 규칙으로 정리한다.

## 검증 기준

자동 검증:

- 저해상도 UI 스모크에서 텍스트 오버플로와 화면 밖 버튼이 없어야 한다.
- 주요 화면 캡처가 검은 화면이 아니어야 한다.
- 기존 게임 진행, 저장, 판단, 리포트 도달 흐름이 깨지지 않아야 한다.

사람 검증:

- 주요 화면을 처음 본 사람이 핵심 행동을 3초 안에 찾는다.
- 심사 화면은 대시보드가 아니라 서류 처리 책상처럼 보인다.
- `신청서`/`소득증명` 라벨이 억지로 끼워진 느낌이 없다.
- 버튼과 패널보다 신청서, 기준표, 도장이 먼저 보인다.
- 캡처만 봐도 같은 게임의 일관된 미감으로 읽힌다.

상용 판단:

- 핵심 화면 평균 4/5 이상
- 심사 화면은 4/5 미만이면 실패
- 텍스트가 박스 안에 들어가는 것만으로는 통과하지 않는다.
- 기능이 많아 보이는 화면보다 플레이 행동이 선명한 화면을 우선한다.

## 첫 구현 티켓

다음 작업자는 아래 티켓부터 시작한다.

1. `심사 화면 캡처 기준선 작성`
   - 현재 1280x720/1920x1080 캡처를 기준으로 핵심/보조/불필요 정보를 표시한다.

2. `서류 영역 재설계`
   - `신청서`/`소득증명`을 작은 라벨 박스가 아닌 문서 탭/캡션 체계로 바꾼다.

3. `도장 트레이 재구성`
   - 결정 버튼 5개를 도장 트레이처럼 보이게 하고, 단축키 표시는 보조화한다.

4. `보조 패널 축약`
   - 추천, 지표, 기록성 문구를 현재 판단에 필요한 정보 중심으로 줄인다.

5. `비주얼 QA 기록`
   - 수정 전후 캡처와 사람 눈 평가 점수를 문서에 남긴다.

## 진행 기록

### 2026-06-08 1차 심사 화면 정리

작업 대상:

- 심사 화면 상단 서류 슬롯
- 하단 결정 버튼/도장 영역

적용 내용:

- `신청서`/`소득증명`을 포함한 서류 슬롯을 작은 라벨 박스 구조에서 문서 탭/캡션 구조로 바꿨다.
- 서류 라벨은 슬롯 하단 장식판에 끼우지 않고, 종이 카드 상단의 탭처럼 같은 폭과 기준선으로 배치했다.
- 결정 버튼 5개는 부모 버튼을 어두운 도장 패드로 쓰고, 실제 도장 이미지를 그 위에 올리는 구조로 바꿨다.
- 단축키 숫자는 도장보다 덜 튀도록 작게 낮췄고, 결정명은 도장 패드 하단 라벨로 정리했다.
- 하단 결정 영역 명칭과 배경을 `Decision Stamp Tray` 쪽으로 바꿔 일반 버튼 묶음보다 도장 트레이처럼 읽히게 했다.

검증:

- QA 도구 Windows 빌드 성공: `Logs/build_windows_qa_visual_direction_review_ui_20260608_01.log`
- 저해상도 UI 스모크 통과: `Logs/runtime_low_resolution_visual_direction_review_ui_20260608_01.log`
- 일반 창 저해상도 UI 스모크 통과: `Logs/runtime_low_resolution_visual_direction_review_ui_visible_20260608_01.log`
- 결과 JSON: `Builds/QA/v0.3.0/low_resolution_ui/care_review_low_resolution_ui_smoke_result.json`
