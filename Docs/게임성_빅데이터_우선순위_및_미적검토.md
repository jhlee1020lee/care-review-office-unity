# 게임성 빅데이터 우선순위 및 미적 검토 절차

작성일: 2026-06-04

## 목적

이 문서는 실제 Steam 출시 준비가 아니라, `돌봄지원 심사소`를 플레이어가 직접 체감하는 게임으로 더 깊고 아름답게 만들기 위한 작업 우선순위를 정리한다.

최근 작업의 Store Presence, 배포 패킷 무결성, 릴리즈 후보 감사는 제출 준비에는 의미가 있지만, 현재 목표에서는 1순위가 아니다. 앞으로는 자동 QA보다 인게임 재미, 데이터 분석 플레이, 시각적 완성도를 먼저 올린다.

## 방향 전환 원칙

1. 플레이어가 보는 화면과 조작 루프를 우선한다.
2. QA/출시 문서는 새 인게임 기능을 검증하는 데 필요한 최소 범위로만 다룬다.
3. "빅데이터 게임"이라는 정체성은 문서나 로그가 아니라, 게임 안에서 데이터 흐름을 읽고 판단하는 경험으로 보여준다.
4. 자동 스모크 테스트 통과보다 실제 실행 화면의 아름다움, 정보 위계, 긴장감, 반복 동기가 더 중요하다.
5. 기능을 추가할 때는 최소 1회 실제 게임 실행 또는 캡처 검토를 포함한다.

## 인게임 기능 우선순위

### P0. 대규모 심사 데이터 대시보드

목표: 플레이어가 개별 사례만 처리하는 것이 아니라, 여러 사례의 흐름과 위험군을 읽는 게임으로 확장한다.

구현 후보:

- 40개 사례 전체의 지원 결정 분포, 위험 점수, 예산 소모, 이의제기 발생률 시각화
- 가족 유형, 소득 구간, 증빙 부족, 돌봄 부담, 긴급도별 필터
- "오늘의 이상치" 패널: 통계적으로 튀는 사례를 자동 추천
- 판단 전/후 지표 변화 그래프
- 데이터 필터를 바꾸면 추천 조사/지원 우선순위가 즉시 바뀌는 UI

게임성 기준:

- 플레이어가 "자료를 분석해서 더 나은 판단을 했다"고 느껴야 한다.
- 단순 표가 아니라, 선택을 유도하는 대시보드여야 한다.
- 한 화면에 너무 많은 수치를 쌓지 않고, 색상/크기/정렬로 위험도를 읽게 해야 한다.

### P1. 정책 시뮬레이터

목표: 예산, 지원 기준, 감사 강도 변경이 결과에 어떤 영향을 주는지 예측하게 한다.

구현 후보:

- 예산 상한 슬라이더
- 긴급 지원 우선/공정 배분/부정수급 억제 같은 정책 프리셋
- 기준 변경 전후 승인율, 위험 방치율, 이의제기율 비교
- "이 정책으로 5일을 진행하면 어떤 엔딩 확률이 올라가는가" 예측 카드

게임성 기준:

- 정책 선택이 단순 옵션이 아니라 딜레마가 되어야 한다.
- 좋은 결과와 부작용을 동시에 보여줘야 한다.
- 플레이어가 다음 회차에서 다른 기준을 시험하고 싶어야 한다.

### P1. 플레이어 결정 패턴 분석

목표: 플레이어의 판단 성향을 데이터로 되돌려 보여준다.

구현 후보:

- 지원 관대함, 감사 엄격함, 긴급도 민감도, 증빙 의존도 지표
- 놓친 위험 신호 목록
- 비슷한 사례에서 일관성이 흔들린 지점 표시
- "당신은 이런 사례에 약하다" 코칭 카드
- 다음 회차 목표와 직접 연결되는 분석 결과

게임성 기준:

- 결과 리포트가 점수표가 아니라 플레이어 성향의 거울처럼 느껴져야 한다.
- 비판만 하지 말고, 다음 플레이 목표를 제안해야 한다.
- 수치가 내러티브 문장과 연결되어야 한다.

### P2. 가상 플레이어/AI 집단 비교

목표: 플레이어의 판단을 여러 운영 성향 에이전트와 비교해 분석 게임의 깊이를 높인다.

구현 후보:

- 균형형, 지원 확대형, 긴축 감사형, 위험 회피형, 예산 절약형 에이전트 비교
- 같은 사례에서 각 에이전트가 어떤 결정을 내렸는지 표시
- 플레이어와 에이전트의 일치율/충돌률
- 특정 에이전트를 기준으로 리플레이 챌린지 제공

게임성 기준:

- "내 선택이 유일한 답이 아니다"라는 감각을 줘야 한다.
- 비교표는 읽기 쉬워야 하며, 다음 행동으로 이어져야 한다.
- 에이전트가 단순 점수 계산기가 아니라 관점처럼 보여야 한다.

### P2. 빅데이터 사건 이벤트

목표: 데이터 변화가 게임 사건으로 발생하게 만든다.

구현 후보:

- 특정 유형 사례 급증
- 예산 압박 경보
- 부정수급 의심 패턴 증가
- 민원/여론 변화
- 지역별 돌봄 공백 급증

게임성 기준:

- 사건은 단순 팝업이 아니라 판단 기준을 흔들어야 한다.
- 플레이어가 대시보드를 다시 열어 원인을 찾게 만들어야 한다.
- 매 회차의 리듬과 긴장감을 만들어야 한다.

### P3. 케이스 군집/트렌드 탐색

목표: 단일 사례 처리에서 벗어나 사례 묶음의 패턴을 탐색하게 한다.

구현 후보:

- 유사 사례 군집 보기
- 지원 결과별 대표 사례
- 오판 가능성이 높은 군집
- 전 회차 대비 위험 신호 변화
- "왜 이 사례가 이 그룹에 속했는가" 설명 패널

게임성 기준:

- 플레이어가 데이터 탐정처럼 패턴을 찾아야 한다.
- 군집 결과가 실제 판단이나 정책 시뮬레이터에 영향을 줘야 한다.
- 시각적으로 복잡해도 조작은 단순해야 한다.

## 미적 검토 절차

자동 QA는 화면이 존재하는지 확인할 수 있지만, 화면이 아름다운지 판단하지 못한다. 따라서 주요 기능 작업마다 실제 게임 화면을 확인한다.

### 1. 실행 전 확인

- 최신 빌드 또는 Unity Play Mode로 진입 가능한지 확인한다.
- 현재 작업 기능까지 도달하는 가장 짧은 경로를 정한다.
- 확인할 화면 목록을 정한다.
- 해상도는 최소 `1920x1080`과 저해상도 1개를 확인한다.

필수 확인 화면:

- 메인 메뉴
- 운영 기준 선택
- 사례 심사 화면
- 데이터/분석 대시보드
- 결과 리포트
- 캠페인 기록
- 사례 자료실
- 새로 구현한 기능 화면

### 2. 캡처 기준

각 화면은 가능하면 스크린샷으로 남긴다.

권장 캡처:

- 전체 화면 기본 상태
- 마우스/포커스가 올라간 상태
- 경고/이벤트/강조 상태
- 데이터 필터 적용 전후
- 결과 리포트 또는 분석 요약 상태

캡처 파일은 `Builds/QA/v0.3.0/visual_review/` 아래에 보관하는 것을 기본으로 한다.

### 3. 미적 평가 체크리스트

각 화면을 1-5점으로 평가한다.

| 항목 | 기준 |
| --- | --- |
| 첫인상 | 화면이 완성된 게임처럼 보이는가 |
| 정보 위계 | 가장 중요한 정보가 먼저 보이는가 |
| 색상 | 위험, 승인, 보류, 데이터 강조 색이 일관되고 아름다운가 |
| 밀도 | 데이터가 많아도 답답하거나 난잡하지 않은가 |
| 글꼴 | Paperlogy 적용 상태가 자연스럽고 가독성이 좋은가 |
| 정렬 | 패널, 표, 버튼, 아이콘이 정돈되어 있는가 |
| 상호작용 | 버튼/필터/탭 상태 변화가 명확한가 |
| 긴장감 | 심사 게임 특유의 압박과 분위기가 느껴지는가 |
| 빅데이터 감각 | 데이터 규모, 추세, 패턴을 시각적으로 느낄 수 있는가 |
| 재플레이 동기 | 다음 회차를 해보고 싶은 목표가 보이는가 |

합격 기준:

- 핵심 화면 평균 4.0 이상
- 새 기능 화면은 `첫인상`, `정보 위계`, `빅데이터 감각`이 모두 4점 이상
- 3점 이하 항목은 다음 작업 후보로 기록

### 4. 실제 플레이 검토 질문

기능 구현 후 직접 10-20분 플레이하며 아래 질문에 답한다.

- 지금 화면은 게임인가, 관리 도구인가?
- 숫자를 읽는 이유가 플레이 결정으로 이어지는가?
- 플레이어가 손해를 감수하고 다른 선택을 해보고 싶어지는가?
- 결과가 납득되면서도 다음 판을 부르는가?
- 화면이 한눈에 보기 좋고, 오래 봐도 피로하지 않은가?
- 데이터가 장식이 아니라 판단의 무기가 되는가?

### 5. 시각 보강 우선순위

미적 검토에서 낮은 점수가 나오면 아래 순서로 고친다.

1. 정보 위계: 가장 중요한 수치/판단/위험 신호를 크게 재배치
2. 레이아웃: 카드 과밀, 표 과밀, 버튼 산만함 정리
3. 색상: 승인/보류/거절/위험/예산 색상 체계 정리
4. 모션/피드백: 필터 변경, 판단 확정, 위험 경보에 반응 추가
5. 데이터 시각화: 막대, 추세, 군집, 비교 그래프 개선
6. 내러티브 연결: 수치 옆에 짧은 해석 문장 추가

## 2026-06-05 초보자 진입 보강 및 수업 요건 재검토

첫 실행 직후 플레이어가 무엇을 해야 할지 모를 수 있다는 지적을 반영해, 최신 소스 기준 초반 진입 문구와 튜토리얼 흐름을 보강했다.

- 새 저장 슬롯/경력 기록이 없는 상태에서는 메인 버튼이 `첫 근무 시작`으로 바뀐다.
- 메인 메뉴 상태 문구가 첫 행동을 직접 안내한다: `처음이면 [첫 근무 시작]을 누르세요. 4장 안내 후 첫 신청서를 함께 처리합니다.`
- 완전 신규 상태에서는 상태 영역을 짧은 `오늘 순서: 안내 확인 -> 문서 읽기 -> 기준 확인 -> 도장 선택` 중심으로 정리한다.
- 첫 실행 고지가 메인 메뉴의 `첫 근무 시작` 버튼과 4장 안내를 연결한다.
- 캠페인 시작 안내를 `오늘 할 일`, `읽는 순서`, `판단 기준`, `첫 사례 시작` 4장으로 정리했다.
- 첫 사례 하이라이트 가이드는 실제 심사 화면의 정보량을 단계적으로 설명하기 위해 5단계를 유지했다.
